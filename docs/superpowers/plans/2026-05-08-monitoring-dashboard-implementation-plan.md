# AI Agent 平台监控看板 — 实施规划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 为风投机构内部 AI Agent 平台搭建三层监控看板，从零建立数据采集体系，分阶段交付可用看板。

**Architecture:** 分四个阶段实施——数据基础建设 → 核心看板（L1+L3） → 行为分析看板（L2） → 高级分析。每阶段交付可用看板，逐步丰富指标。

**关联设计文档:** `docs/superpowers/specs/2026-05-07-ai-platform-monitoring-dashboard-design.md`

---

## 实施原则

1. **价值优先**：先交付高管最关心的指标（渗透率、活跃度），再逐步深入
2. **数据先行**：埋点必须先于看板开发，确保数据积累
3. **分阶段交付**：每阶段产出可用看板，不等全部完成才上线
4. **最小埋点原则**：每阶段只接入该阶段看板所需的埋点，避免过度工程

---

## 阶段总览

| 阶段 | 内容 | 预期产出 | 建议时长 |
|------|------|----------|----------|
| Phase 0 | 数据基础建设 | 事件采集 SDK + 数据管道 + 存储 | 2-3 周 |
| Phase 1 | L1 高管概览 + L3 系统观测 | 第一版可用看板 | 2-3 周 |
| Phase 2 | L2 行为分析 | 模块级分析看板 | 3-4 周 |
| Phase 3 | 高级分析 + 告警 | 分群迁移、主题聚类、自动告警 | 2-3 周 |

**总计：9-13 周**（可根据团队资源并行压缩）

## Phase 0：数据基础建设（第 1-3 周）

### 目标

建立统一的事件采集、传输、存储体系，为所有看板指标提供数据源。

### Task 0.1：定义事件规范

- [ ] **制定事件命名与字段规范**

  统一所有埋点事件的命名规则、字段类型、必填/选填约束：

  ```
  事件命名：{module}_{action}（小写 + 下划线）
  通用字段（所有事件必带）：
    - event_id: string (UUID)
    - event_name: string
    - timestamp: ISO8601
    - user_id: string
    - group: enum (智能应用组|医疗健康组|内容金融组|投资组合管理组|数字化平台组|技术企业组|消费组|全球娱乐组)
    - module: enum (chat|deepseek|ic_memo)
    - session_id: string (nullable)
  ```

- [ ] **输出《事件埋点规范文档》**

  包含：每个事件的名称、触发条件、全部字段定义、字段示例值、版本号

- [ ] **与前后端开发团队对齐规范**

### Task 0.2：埋点 SDK / 工具选型

- [ ] **评估方案**

  | 方案 | 适用场景 | 优缺点 |
  |------|----------|--------|
  | 自研轻量 SDK | 对数据完全掌控、事件少 | 灵活但维护成本高 |
  | 开源方案（Segment/Rudderstack 自部署） | 中等规模、需多端采集 | 功能全但部署复杂 |
  | 后端直接写入 | 事件主要在服务端产生 | 最简单，前端行为需额外处理 |

- [ ] **确定最终方案并完成技术选型文档**

  考虑因素：
  - 平台现有技术栈兼容性
  - 是否需要前端（浏览器端）埋点（复制、放弃等行为需要前端）
  - 数据存储选择（时序数据库 / OLAP 数据库 / 数据湖）
  - 看板工具选择（Metabase / Grafana / Superset / 自研）

### Task 0.3：实现核心事件采集

- [ ] **第一批埋点（Phase 1 所需）**

  优先实现以下事件，覆盖 L1 + L3 所有指标：

  | 事件 | 采集位置 | 优先级 |
  |------|----------|--------|
  | session_start | 后端 | P0 |
  | session_end | 后端 + 前端辅助（检测放弃） | P0 |
  | message_send | 后端 | P0 |
  | message_receive | 后端 | P0 |
  | feedback (👍/👎) | 前端 → 后端 | P0 |
  | regenerate | 前端 → 后端 | P0 |
  | copy | 前端 | P1 |
  | file_upload | 后端 | P0 |
  | security_block | 后端 | P0 |

- [ ] **验证数据链路**

  端到端测试：触发事件 → 传输 → 入库 → 可查询

### Task 0.4：数据管道与存储

- [ ] **搭建数据管道**

  ```
  应用层埋点 → 消息队列（可选） → 数据清洗 → 存储层
  ```

- [ ] **设计数据表/模型**

  核心表：
  - `events`：原始事件流（append-only）
  - `sessions`：会话维度聚合（session_id, user_id, group, module, model, start_time, end_time, message_count, total_tokens, end_reason）
  - `users`：用户维度（user_id, group, first_active_date, last_active_date, total_sessions）
  - `daily_metrics`：预聚合日度指标（date, group, dau, sessions, messages, tokens）

- [ ] **确认数据保留策略**

  - 原始事件：保留 12 个月
  - 聚合数据：永久保留
  - 个人明细：脱敏规则（如需）

### Task 0.5：用户元数据管理

- [ ] **建立用户-部门映射表**

  ```
  users_meta:
    - user_id: string
    - name: string
    - group: enum (8 个部门)
    - role: string
    - register_date: date
    - status: enum (active|inactive)
  ```

- [ ] **确定数据来源**

  从 HR 系统/企业通讯录同步，还是平台注册时录入？确定同步频率。

- [ ] **确定总员工数的维护方式**

  渗透率计算需要各组总人数作为分母，需要一个可维护的配置或同步机制。

## Phase 1：L1 高管概览 + L3 系统观测（第 3-6 周）

### 目标

交付第一版可用看板，高管可以看到核心使用数据；平台团队可以看到系统健康状态。

### Task 1.1：L1 渗透率与活跃度面板

- [ ] **实现指标计算逻辑**

  | 指标 | 计算方式 |
  |------|----------|
  | DAU | COUNT(DISTINCT user_id) WHERE date = today |
  | WAU | COUNT(DISTINCT user_id) WHERE date >= today - 7 |
  | MAU | COUNT(DISTINCT user_id) WHERE date >= today - 30 |
  | 全员渗透率 | MAU / config.total_employees |
  | 分组渗透率 | group_mau / config.group_size[group] |

- [ ] **搭建看板页面/仪表盘**

  - 顶部：DAU / WAU / MAU 大数字卡片 + 环比箭头
  - 中部：全员渗透率 + 分组渗透率柱状图（8 个部门）
  - 底部：活跃度趋势折线（30 天）

- [ ] **实现时间筛选器**（7天 / 30天 / 90天 / 自定义）

- [ ] **实现部门筛选器**（全部 / 单选某组）

### Task 1.2：L1 留存模块

- [ ] **实现留存计算逻辑**

  ```
  W1 留存：首次使用日期为 D 的用户中，D+7 天内再次活跃的比例
  W4 留存：首次使用日期为 D 的用户中，D+28 天内再次活跃的比例
  W12 留存：首次使用日期为 D 的用户中，D+84 天内再次活跃的比例
  沉默用户：last_active_date < today - 30 的用户数 / 总注册数
  ```

- [ ] **搭建留存曲线图**

  - 支持全员 / 按组切换
  - 按周 cohort 展示留存曲线
  - 沉默用户比例卡片 + 趋势

- [ ] **注意**：W12 需要 12 周数据积累，初期此指标为空，标注"数据积累中"

### Task 1.3：L1 使用深度模块

- [ ] **实现指标计算**

  | 指标 | 计算 |
  |------|------|
  | 人均会话数 | total_sessions / active_users (时间段内) |
  | 人均消息数 | total_messages / active_users |
  | 平均会话长度 | AVG(session.message_count) |
  | 多功能交叉使用率 | AVG(COUNT(DISTINCT module) per user) |
  | 高/中/低频分布 | 按用户活跃天数分桶 |

- [ ] **搭建看板组件**

  - 数字卡片：人均会话数、人均消息数、平均会话长度
  - 分布直方图：会话长度分布
  - 堆叠面积图：高/中/低频用户占比（按月趋势）

### Task 1.4：L1 回答质量模块

- [ ] **实现指标计算**

  | 指标 | 计算 |
  |------|------|
  | 好评率 | thumbs_up / (thumbs_up + thumbs_down) |
  | 重新生成率 | regenerate_count / total_responses |
  | 会话放弃率 | sessions WHERE end_reason='abandon' / total_sessions |
  | 复制率 | copy_events / total_responses |
  | 高轮数占比 | sessions WHERE message_count > 10 / total_sessions |

- [ ] **搭建看板组件**

  - 好评率大数字 + 趋势线
  - 点踩原因分布饼图
  - 重新生成率、放弃率、复制率数字卡片
  - 高轮数占比 + 阈值标线

- [ ] **前端实现反馈 UI**

  - 每条 AI 回复下方：👍 / 👎 按钮
  - 👎 时弹出原因选择（预设标签：不准确 / 不相关 / 太长 / 格式差 / 其他）

### Task 1.5：L1 功能与模型分布模块

- [ ] **实现指标计算**

  - 功能使用占比：按 module 字段聚合 session 数
  - 模型调用分布：按 model 字段聚合，统计调用次数 + token 消耗
  - 分组×模型交叉：按 (group, model) 二维聚合

- [ ] **搭建看板组件**

  - 功能使用占比环形图
  - 模型调用量堆叠柱状图（按日/周切换）
  - 模型偏好趋势多线折线图
  - 分组×模型热力图
  - Token 消耗总量 + 按功能拆分堆叠柱状图

### Task 1.6：L3 系统观测面板

- [ ] **接入系统指标数据源**

  | 数据源 | 指标 | 接入方式 |
  |--------|------|----------|
  | API Gateway 日志 | 请求量、延迟、状态码 | 日志解析或 APM 集成 |
  | 模型服务 | TTFT、端到端时长、token 吞吐 | 应用内埋点（message_receive 事件已含 latency/ttft） |
  | DeepSeek 实例 | GPU/内存/队列 | 基础设施监控（Prometheus + GPU exporter） |

- [ ] **搭建系统观测看板**

  - API 可用性：各模型 uptime 数字 + SLA 标线（99%）
  - 响应时间：P50/P95/P99 按模型展示
  - TTFT：首 token 延迟趋势线
  - 错误率：趋势线 + 突增告警标记
  - 错误类型分布饼图
  - 并发会话数时间序列
  - DeepSeek 实例负载仪表盘

### Task 1.7：L3 安全合规面板

- [ ] **搭建合规统计看板**

  - 外部模型拦截次数：大数字 + 趋势线
  - 拦截触发用户/部门排行榜
  - 拦截场景分布饼图（文件 vs 数据源）
  - 内容审核拦截次数 + 趋势
  - 敏感字段识别次数 + 趋势

### Task 1.8：L1 通用交互与异常标记

- [ ] **实现趋势异常标记**

  规则：任何核心指标环比下降超过 20% → 自动标红 + 标注

- [ ] **实现活跃度热力图**

  按"星期几 × 小时"展示使用密度

- [ ] **Phase 1 验收**

  - 高管可查看 L1 全部指标
  - 平台团队可查看 L3 系统健康 + 合规统计
  - 数据延迟 < 1 小时（准实时）

## Phase 2：L2 行为分析（第 6-10 周）

### 目标

按功能模块深入拆解用户行为，提供优化线索。

### Task 2.1：补充第二批埋点

- [ ] **新增事件（Phase 2 所需）**

  | 事件 | 采集位置 | 用途 |
  |------|----------|------|
  | memo_generate_start | 后端 | IC Memo 分析 |
  | memo_generate_complete | 后端 | IC Memo 时长分析 |
  | memo_edit | 前端 → 后端 | 编辑行为分析 |
  | memo_section_rating | 前端 → 后端 | 章节质量评分 |
  | memo_download | 前端 → 后端 | 任务完成度 |
  | session_topic | 后端（异步计算） | 主题聚类 |

- [ ] **前端实现主动评分 UI**

  - IC Memo 生成完成后：弹出整体评分（1-5 星）
  - 各章节旁：可选评分（好/一般/差）
  - 长会话（>10 轮）结束时：弹出满意度评分

### Task 2.2：通用对话分析面板

- [ ] **实现指标与图表**

  - 模型选择分布（Claude vs ChatGPT）：饼图 + 趋势
  - 上下文长度分布：直方图（input token 分桶）
  - 对话轮数分布：1-3 / 4-10 / 10+ 堆叠柱状图
  - 长会话占比：数字 + 趋势
  - 复制率：按模型拆分对比
  - 时段分布：小时×星期几 热力图

- [ ] **会话主题分布（异步计算）**

  - 方案：定期（每日）对新增会话做关键词提取或 embedding 聚类
  - 展示：词云 或 Top 10 主题列表 + 各主题会话数
  - 标注：此功能可延后至 Phase 3，取决于实现复杂度

### Task 2.3：私有化模型（DeepSeek）分析面板

- [ ] **实现指标与图表**

  - 敏感文件处理量：数字 + 日趋势
  - 文件类型分布：饼图
  - 模型选择路径：主动选择 vs 系统自动路由 占比
  - 响应满意度：DeepSeek 的 👍/👎 率 vs 公有模型对比

### Task 2.4：IC Memo Agent 分析面板

- [ ] **实现指标与图表**

  - 使用人数与频次：活跃用户数 + 人均生成份数
  - 上传文件类型分布：饼图
  - 文件解析成功率：数字 + 趋势（低于 95% 标红）
  - 生成时长：P50/P95 + 趋势
  - 各章节生成时长：分章节柱状图
  - 用户编辑行为：手动编辑 / 对话调整 / 未修改 三分饼图
  - 章节级编辑率：各章节被修改比例柱状图
  - 章节级质量评分：各章节平均分雷达图
  - 下载率：数字 + 趋势

- [ ] **IC 会议对接（需评估可行性）**

  - 如投决系统有 API：对接 memo_to_ic 事件
  - 如无 API：暂用人工标记或 CSV 导入
  - 此指标可标注为"对接中"，不阻塞其他工作

### Task 2.5：用户分群与迁移面板

- [ ] **实现分群逻辑**

  ```
  高频用户：过去 30 天内 ≥20 天活跃
  中频用户：过去 30 天内 5-19 天活跃
  低频用户：过去 30 天内 1-4 天活跃
  沉默用户：过去 30 天内 0 天活跃
  ```

- [ ] **实现分群迁移桑基图**

  - 每月初计算上月各分群用户流向
  - 展示：桑基图（高→中、中→低、低→沉默、沉默→低 等流向）

- [ ] **实现新用户激活漏斗**

  ```
  注册 → 首次使用（当天） → 第二次使用（7天内） → 周活跃（连续2周活跃）
  ```
  展示：漏斗图 + 各步骤转化率

- [ ] **实现功能探索路径**

  - 统计用户首次使用的功能 → 第二个使用的功能 → 第三个
  - 展示：桑基图或流向图

- [ ] **Phase 2 验收**

  - 管理团队可按模块查看详细行为分析
  - IC Memo 的编辑行为和章节质量有数据
  - 分群迁移路径可视化

## Phase 3：高级分析 + 告警（第 10-13 周）

### 目标

补全高级分析能力（主题聚类、异常检测）和自动化告警通知。

### Task 3.1：会话主题聚类（如 Phase 2 未完成）

- [ ] **选择实现方案**

  | 方案 | 复杂度 | 效果 |
  |------|--------|------|
  | 关键词提取（TF-IDF） | 低 | 粗粒度，适合 MVP |
  | Embedding 聚类（调用模型做 embedding → K-means） | 中 | 语义级别，更准确 |
  | LLM 摘要打标（用小模型为每个会话打 1-3 个标签） | 中高 | 最直观，但有成本 |

- [ ] **实现每日异步计算任务**

  - 对前一天新增会话进行主题标注
  - 结果写入 `session_topics` 表

- [ ] **搭建主题分布图表**

  - Top 10 主题 + 会话数
  - 主题趋势变化（新兴话题标记）
  - 按部门的主题差异对比

### Task 3.2：告警系统

- [ ] **实现告警规则引擎**

  | 级别 | 规则 | 通知渠道 |
  |------|------|----------|
  | Critical | API 可用性 < 99% | 即时推送（飞书/钉钉 webhook） |
  | Critical | 错误率 > 5%（5分钟窗口） | 即时推送 |
  | Warning | P95 响应时间 > 5s | 日报汇总 |
  | Warning | 错误率 > 2%（1小时窗口） | 日报汇总 |
  | Info | 沉默用户比例环比上升 > 10% | 周报 |
  | Info | 某模型调用占比周环比变化 > 30% | 周报 |

- [ ] **实现通知推送**

  - 即时通知：飞书/钉钉 Bot webhook
  - 日报：每日 9:00 推送前一日异常摘要
  - 周报：每周一推送上周关键指标汇总

### Task 3.3：自动化周报生成

- [ ] **设计周报模板**

  ```
  【AI 平台周报 - W{week_number}】

  📊 核心指标
  - 本周活跃用户：XX 人（环比 +X%）
  - 全员渗透率：XX%
  - 人均会话数：X.X

  📈 亮点
  - {自动识别环比增长最大的指标}

  ⚠️ 关注
  - {自动识别环比下降最大的指标}
  - {沉默用户变化}

  🏆 部门排行
  - 最活跃：{group}（渗透率 XX%）
  - 增长最快：{group}（环比 +XX%）
  ```

- [ ] **实现自动推送**

  每周一早上自动生成并推送至管理层群组

### Task 3.4：L1 → L2 下钻交互

- [ ] **实现看板层级间导航**

  - L1 任意指标卡片可点击，跳转到 L2 对应模块
  - 示例：点击"IC Memo 使用占比" → 跳转 L2 IC Memo 面板
  - L2 面板可跳转 L3 查看系统级数据

- [ ] **Phase 3 验收**

  - 会话主题自动聚类并展示
  - 告警系统正常触发并推送
  - 周报自动生成并发送
  - 三层看板间下钻交互流畅

| 用户-部门映射数据源 | 渗透率、分组指标 | 中 | 优先确认 HR 系统/通讯录对接方式 |
| 前端埋点工作量 | 复制率、放弃率、编辑行为 | 低 | 可安排前端开发并行 |
| DeepSeek 实例监控接入 | L3 私有化负载 | 低 | 运维团队协助 |
| 投决系统 API | IC Memo 最终价值指标 | 高 | 先标注"对接中"，不阻塞其他工作 |
| 看板工具选型 | 全部可视化 | 中 | Phase 0 确定，建议评估 Metabase/Grafana/Superset |

### 关键风险

1. **数据冷启动**：留存指标（W4/W12）需要时间积累，初期标注"数据积累中"
2. **会话放弃检测**：前端需要实现离开检测（页面关闭/切换），可能有技术难度
3. **主题聚类质量**：自动聚类效果依赖方案选择，建议 Phase 3 再实现，MVP 阶段可人工抽检
4. **IC 会议对接**：取决于投决系统是否有可用 API，优先确认可行性

## 优先级排序（如资源有限）

如果需要在有限资源下做取舍，按以下优先级：

### 必须有（P0）
- 渗透率（全员 + 分组）
- DAU/WAU/MAU
- 人均会话数/消息数
- 模型调用分布
- Token 消耗
- API 可用性 + 错误率
- 外部模型拦截统计

### 应该有（P1）
- 留存曲线（W1/W4/W12）
- 好评率 + 重新生成率
- 高/中/低频用户分布
- IC Memo 使用人数、生成时长、下载率
- 响应时间 P50/P95

### 可以后做（P2）
- 会话放弃率（前端检测复杂度高）
- 会话主题聚类（需要额外计算资源）
- 分群迁移桑基图
- 章节级编辑率和质量评分
- 自动化周报

### 可以不做（P3）
- IC Memo → IC 会议对接（强依赖外部系统）
- 功能探索路径（需长期数据积累）
